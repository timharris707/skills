## 1. Verdict

## Per-question rulings

**Q1 — Edit identity: CAUTION (titles are fine, but not as-is).** Go with (a) titles to avoid a second schema evolution in two releases — that trade-off is correctly judged. But the strawman's premise ("titles are stable within a run") conflates *stable* with *unique*, and the code shows they are not unique: `board_verdict.py:45` defines the finding containers with no uniqueness constraint, `_finding_titles()` (`:525-532`) just appends every title, and `amend --on` (`:593-597`) resolves a duplicate by **silently matching the first occurrence** (`in` membership at `:594`). A synthesizer merging model output can absolutely emit two concerns with the same title. If `changes.json.edits[].resolves[]` keys by `{list, title}` and a title is duplicated, the join is ambiguous *and* the Q3 completeness check ("every blocker resolved-or-unresolved") is unsound — it can be satisfied against the wrong instance. Ruling: keep titles, but make the conductor **guarantee title-uniqueness-within-a-list at synthesis** (disambiguate or refuse), and have the `changes` validator reject an ambiguous `resolves`. This also retro-fixes a latent `amend` bug for free.

**Q2 — Mapping location: CAUTION — prefer (a), reject the (b) strawman.** The pointer-in-`verdict.json` inverts the tool's own provenance direction and its write discipline. `previous_run` is written on the **new** run's verdict pointing **back** to the old run's sha (`revise.py:300, 328-332`; validated `board_verdict.py:146-168`) — downstream artifact → upstream artifact. The brief's (b) puts a pointer in `verdict.json` (upstream) pointing **forward** to `changes.json` (downstream): backwards. Worse, it reopens a finalized judgment record. `cli.py:554-613` is explicit that failure paths must not corrupt prior good state ("the stale verdict.json … is unlinked HERE — never higher up — so an unexpected exception above doesn't destroy a prior good state"), and `amend` guards `verdict.json` with an optimistic-sha check (`:622-625`) on the assumption it owns rewrites. With (a), "a revision failure cannot corrupt `verdict.json`" is **structurally true** because the revision step never opens it; with (b) it's a property you must carefully preserve. The (b) benefit — "verifiable from the verdict alone" — is largely illusory: the pointer gives you a sha but not the bytes, so you still need `changes.json` to verify anything. Ruling: (a) file-only; put a `verdict_sha256` reverse-pointer **inside `changes.json`** (matching `previous_run`'s direction), and surface "this run produced an endorsed revised-draft (sha …)" on the run card / consensus render for discoverability. Note the plumbing already half-exists: `changes` is in `LIFECYCLE_KEYS` (`synthesizer.py:195-203`) so the model can't fabricate it — but that argues only that (b) is *feasible*, not that it's *right*.

**Q3 — `changes.json` shape: CAUTION.** The line-range-locator-plus-two-shas approach is correct and embedded before/after text should **not** be added — it would duplicate what the mechanical diff derives and create a second source of truth. But the schema's load-bearing integrity property is left implicit and must be made explicit: **the `edits[]` must reconcile with the mechanical `original→revised` diff** the conductor computes anyway (`get_opcodes`, per Q8). Every diff hunk must map to ≥1 edit entry and every edit locator must fall within a real hunk; otherwise "here is what I changed and why" is an unverifiable model assertion, not a fact about the bytes. Second: the completeness check is **structural, not semantic** — it proves nothing was silently dropped, not that any fix is adequate (that's the endorsement pass's job); state this honestly. Third, a factual fix: **the third finding container is `dissent[]` (with a `who` field), not `caveats[]`** as the brief repeats (`board_verdict.py:45` `EVIDENCE_CONTAINERS = ("blockers", "dissent", "concerns")`; no `caveats[]` finding list exists, and `amend`'s join surface is blocker/concern only). A `dissent` is a recorded *seat disagreement about judgment*, not a source defect — a fixed copy cannot "resolve" it. Scope `resolves` (and the completeness gate) to **blockers + concerns**, matching `amend`'s join surface; treat dissent as commentary. Model-authored fields limited to `summary`/`resolves`/`note` with the conductor computing `n`, locators-validation, shas, statuses is exactly right (skeleton-merge discipline, `synthesizer.py:459-485`).

**Q4 — Prose vs code decision: SHIP (one clarification).** (c) heuristic + override + recorded-resolved-value is the right call and matches the recipe-replay discipline (`recipe.py:240`); the failure mode of a `.py` redlined as prose is visible and recoverable. One gap: **extensionless files** (`Makefile`, `Dockerfile`, `LICENSE`) must fall into the unknown→refuse branch, not silently into prose — "no recognized extension" and "unknown extension" should route identically to "pass `--source-type`." With that, ship.

**Q5 — One spawn or two: CAUTION.** One spawn (a) is right for keeping mapping and text co-authored, but the strawman waves off its worst failure: a **silently truncated-but-still-parseable** revised block. The seat emits the *entire* revised source plus the JSON; for a large source that reply approaches the model's output ceiling, and a reply cut at 60% yields a `revised-draft.md` missing its tail with a clean-looking `changes.json` — shipped as "the board's fixed copy." Retry-on-`invalid` (`synthesizer.py:604-619`, `spawn.py:172`) only catches *unparseable*, not *truncated*. Two mandatory mitigations: (1) the **Q3 diff-reconciliation check doubles as truncation detection** — a truncated tail appears as a large trailing deletion hunk with no corresponding edit, which fails reconciliation and forces the reject path; (2) a **source-size preflight** that refuses loudly when the source is large enough that full-text + JSON risks the output cap. With those, (a) + the mirrored failure artifacts (`revised-draft-rejected.md` + `changes-rejected.json` + warning + exit 0, `--strict-exit`→4) is correct.

**Q6 — Endorsement pass: SHIP on ON-by-default (i); CAUTION on failure posture.** ON-by-default with `--no-endorse` is correct — un-endorsed, "board-endorsed" is marketing and R4 bites; the +40–60% spawn cost has an opt-out. But per-edit tokenized replies are parse-fragile at scale (15 edits × N seats), and the strawman doesn't state the failure posture. It must be explicit: **a missing/unparseable per-edit position records as `ABSTAIN`, never `ENDORSE`** — defaulting silence to endorsement manufactures the exact consensus the feature exists to earn. The render must visibly distinguish "endorsed" from "abstained/silent." Second gap: seats can endorse/object to `edits` but not to `unresolved` entries — yet a seat's most important objection may be "you should *not* have left this unresolved." Extend the endorsement surface to cover `unresolved` too (this is what makes Q7 safe).

**Q7 — Conflict posture: SHIP (with the Q6 tie-in).** (c) apply-non-conflicting-and-surface-`unresolved` matches the tool's degrade-loudly-never-discard posture everywhere else, and exit-code neutrality for `unresolved`-nonempty is right (exit codes are about pipeline integrity, not content, consistent with `cli.py:554-613`). The one exposure: "these two findings conflict" is a *model assertion*, and a lazy revision seat can launder "I didn't do the hard edit" as "conflict." That is only caught if the endorsement pass can object to `unresolved` entries — so Q7's soundness **depends on** the Q6 extension. With that link, ship.

**Q8 — Redline artifacts: CAUTION (two changes).** Requiring a synthesis path (refuse `--output revised-draft` without `verdict.json`) is correct — the revision brief *is* "resolve these findings," and improvising from raw reviews would put a model where the conductor's skeleton belongs. Rejecting CriticMarkup is fine. But: (1) **prose should also emit a `unified_diff` `.patch`** — it's free (`revise.py:259` already uses `unified_diff`), it's the standard review/apply mechanism, and it removes the reliance on the novel word-level HTML redline as the only "diff view." The brief is internally inconsistent here: its prose artifact set has no patch, yet it justifies dropping a text redline by saying "the … unified diff cover[s] … applying." (2) The word-level `<ins>/<del>` HTML redline is **genuinely new machinery** (`get_opcodes` confirmed unused; `render_handoff.py:217-286` is pure `{{TOKEN}}`/`{block}` substitution) — build it as a **separate Python function that emits a pre-rendered HTML string injected as one token/block** (do *not* teach the substitution engine to diff — mirror how `delta_*` slots already inject rendered lists), sequence it **last**, and **do not gate release on it**. ## 4. Invariants and guardrails

- **INV-1 (integrity spine):** `changes.json.edits[]` must reconcile 1:1-or-better with the mechanical `original→revised` diff. Every diff hunk ↔ ≥1 edit; every locator ⊂ a hunk. Reconciliation failure ⇒ reject path (retry, then `*-rejected` + exit 0 / `--strict-exit` 4). This one invariant also makes truncation a loud failure.
- **INV-2:** the revision step **never opens `verdict.json`** (Q2a). A revision failure is then structurally incapable of corrupting the judgment record — matching the `cli.py:554-613` "don't destroy prior good state" discipline.
- **INV-3:** finding titles are unique within a list before any `resolves` join exists.
- **INV-4:** endorsement silence ⇒ `ABSTAIN`, rendered distinctly from `ENDORSE`. No model reply is ever *interpreted* as endorsement.
- **INV-5:** model authorship stays confined to `summary`/`resolves`/`note`; conductor owns `n`, shas, statuses, locator validation, provenance (`PROTECTED_SKELETON_KEYS` + `LIFECYCLE_KEYS`, `synthesizer.py:192-203`).
- **INV-6 (D5):** no-flag default run stays byte-identical; assert with the existing golden fixtures plus a new "revision path absent ⇒ zero new bytes" test.
- **INV-7 (D6):** source is never written; the revision step has no write path to the source, only to the run dir.
