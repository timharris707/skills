Claude seat review — v1.13 fix-it revision artifact (Architecture & systems)

I verified the brief's five "facts" and the reusable substrate against the code at the cited paths. The substrate claims are all true (synthesizer seat-choice, two-attempt retry, protected-key skeleton merge, extract_json_object, DATA-fence neutralizer, --output decorative, get_opcodes genuinely unused, pure-substitution HTML). Two brief claims are wrong or incomplete, and both change rulings. The design is architecturally sound and correctly patterned on the synthesizer — but three of its strawmen ship a real integrity hazard unless specific guardrails, currently left implicit, are made mandatory.

1. Verdict

CAUTION — proceed only with the changes below. Confidence: high (both fact-checks returned; every objection is code-backed with citations, not judgment calls). What would flip it to SHIP: adopt Q2(a) file-only (don't reopen verdict.json), make the diff-reconciliation check (Q3/Q5) a release-gating invariant, add title-uniqueness enforcement (Q1), default unparseable endorsement tokens to ABSTAIN (Q6), and reconcile the caveats vs dissent container error (Q3). None of these are "don't build it" — they are "build it slightly differently."

Per-question rulings

Q1 — Edit identity: CAUTION (titles are fine, but not as-is). Go with (a) titles to avoid a second schema evolution in two releases — that trade-off is correctly judged. But the strawman's premise ("titles are stable within a run") conflates stable with unique, and the code shows they are not unique: board_verdict.py:45 defines the finding containers with no uniqueness constraint, _finding_titles() (:525-532) just appends every title, and amend --on (:593-597) resolves a duplicate by silently matching the first occurrence (in membership at :594). A synthesizer merging model output can absolutely emit two concerns with the same title. If changes.json.edits[].resolves[] keys by {​list, title} and a title is duplicated, the join is ambiguous and the Q3 completeness check ("every blocker resolved-or-unresolved") is unsound — it can be satisfied against the wrong instance. Ruling: keep titles, but make the conductor guarantee title-uniqueness-within-a-list at synthesis (disambiguate or refuse), and have the changes validator reject an ambiguous resolves. This also retro-fixes a latent amend bug for free.

Q2 — Mapping location: CAUTION — prefer (a), reject the (b) strawman. The pointer-in-verdict.json inverts the tool's own provenance direction and its write discipline. previous_run is written on the new run's verdict pointing back to the old run's sha (revise.py:300, 328-332; validated board_verdict.py:146-168) — downstream artifact → upstream artifact. The brief's (b) puts a pointer in verdict.json (upstream) pointing forward to changes.json (downstream): backwards. Worse, it reopens a finalized judgment record. cli.py:554-613 is explicit that failure paths must not corrupt prior good state ("the stale verdict.json … is unlinked HERE — never higher up — so an unexpected exception above doesn't destroy a prior good state"), and amend guards verdict.json with an optimistic-sha check (:622-625) on the assumption it owns rewrites. With (a), "a revision failure cannot corrupt verdict.json" is structurally true because the revision step never opens it; with (b) it's a property you must carefully preserve. The (b) benefit — "verifiable from the verdict alone" — is largely illusory: the pointer gives you a sha but not the bytes, so you still need changes.json to verify anything. Ruling: (a) file-only; put a verdict_sha256 reverse-pointer inside changes.json (matching previous_run's direction), and surface "this run produced an endorsed revised-draft (sha …)" on the run card / consensus render for discoverability. Note the plumbing already half-exists: changes is in LIFECYCLE_KEYS (synthesizer.py:195-203) so the model can't fabricate it — but that argues only that (b) is feasible, not that it's right.

Q3 — changes.json shape: CAUTION. The line-range-locator-plus-two-shas approach is correct and embedded before/after text should not be added — it would duplicate what the mechanical diff derives and create a second source of truth. But the schema's load-bearing integrity property is left implicit and must be made explicit: the edits[] must reconcile with the mechanical original→revised diff the conductor computes anyway (get_opcodes, per Q8). Every diff hunk must map to ≥1 edit entry and every edit locator must fall within a real hunk; otherwise "here is what I changed and why" is an unverifiable model assertion, not a fact about the bytes. Second: the completeness check is structural, not semantic — it proves nothing was silently dropped, not that any fix is adequate (that's the endorsement pass's job); state this honestly. Third, a factual fix: the third finding container is dissent[] (with a who field), not caveats[] as the brief repeats (board_verdict.py:45 EVIDENCE_CONTAINERS = ("blockers", "dissent", "concerns"); no caveats[] finding list exists, and amend's join surface is blocker/concern only). A dissent is a recorded seat disagreement about judgment, not a source defect — a fixed copy cannot "resolve" it. Scope resolves (and the completeness gate) to blockers + concerns, matching amend's join surface; treat dissent as commentary. Model-authored fields limited to summary/resolves/note with the conductor computing n, locators-validation, shas, statuses is exactly right (skeleton-merge discipline, synthesizer.py:459-485).

Q4 — Prose vs code decision: SHIP (one clarification). (c) heuristic + override + recorded-resolved-value is the right call and matches the recipe-replay discipline (recipe.py:240); the failure mode of a .py redlined as prose is visible and recoverable. One gap: extensionless files (Makefile, Dockerfile, LICENSE) must fall into the unknown→refuse branch, not silently into prose — "no recognized extension" and "unknown extension" should route identically to "pass --source-type." With that, ship.

Q5 — One spawn or two: CAUTION. One spawn (a) is right for keeping mapping and text co-authored, but the strawman waves off its worst failure: a silently truncated-but-still-parseable revised block. The seat emits the entire revised source plus the JSON; for a large source that reply approaches the model's output ceiling, and a reply cut at 60% yields a revised-draft.md missing its tail with a clean-looking changes.json — shipped as "the board's fixed copy." Retry-on-invalid (synthesizer.py:604-619, spawn.py:172) only catches unparseable, not truncated. Two mandatory mitigations: (1) the Q3 diff-reconciliation check doubles as truncation detection — a truncated tail appears as a large trailing deletion hunk with no corresponding edit, which fails reconciliation and forces the reject path; (2) a source-size preflight that refuses loudly when the source is large enough that full-text + JSON risks the output cap. With those, (a) + the mirrored failure artifacts (revised-draft-rejected.md + changes-rejected.json + warning + exit 0, --strict-exit→4) is correct.

Q6 — Endorsement pass: SHIP on ON-by-default (i); CAUTION on failure posture. ON-by-default with --no-endorse is correct — un-endorsed, "board-endorsed" is marketing and R4 bites; the +40–60% spawn cost has an opt-out. But per-edit tokenized replies are parse-fragile at scale (15 edits × N seats), and the strawman doesn't state the failure posture. It must be explicit: a missing/unparseable per-edit position records as ABSTAIN, never ENDORSE — defaulting silence to endorsement manufactures the exact consensus the feature exists to earn. The render must visibly distinguish "endorsed" from "abstained/silent." Second gap: seats can endorse/object to edits but not to unresolved entries — yet a seat's most important objection may be "you should not have left this unresolved." Extend the endorsement surface to cover unresolved too (this is what makes Q7 safe).

Q7 — Conflict posture: SHIP (with the Q6 tie-in). (c) apply-non-conflicting-and-surface-unresolved matches the tool's degrade-loudly-never-discard posture everywhere else, and exit-code neutrality for unresolved-nonempty is right (exit codes are about pipeline integrity, not content, consistent with cli.py:554-613). The one exposure: "these two findings conflict" is a model assertion, and a lazy revision seat can launder "I didn't do the hard edit" as "conflict." That is only caught if the endorsement pass can object to unresolved entries — so Q7's soundness depends on the Q6 extension. With that link, ship.

Q8 — Redline artifacts: CAUTION (two changes). Requiring a synthesis path (refuse --output revised-draft without verdict.json) is correct — the revision brief is "resolve these findings," and improvising from raw reviews would put a model where the conductor's skeleton belongs. Rejecting CriticMarkup is fine. But: (1) prose should also emit a unified_diff .patch — it's free (revise.py:259 already uses unified_diff), it's the standard review/apply mechanism, and it removes the reliance on the novel word-level HTML redline as the only "diff view." The brief is internally inconsistent here: its prose artifact set has no patch, yet it justifies dropping a text redline by saying "the … unified diff cover[s] … applying." (2) The word-level <ins>/<del> HTML redline is genuinely new machinery (get_opcodes confirmed unused; render_handoff.py:217-286 is pure {​{​TOKEN}}/{​block} substitution) — build it as a separate Python function that emits a pre-rendered HTML string injected as one token/block (do not teach the substitution engine to diff — mirror how delta_* slots already inject rendered lists), sequence it last, and do not gate release on it. The applyable value lives in revised-draft.* + .patch + changes.json; the redline is a reading aid.

2. Strongest objections (ranked)

  1. Silent truncation ships a corrupted "board-endorsed" copy (Q5). The single highest-severity failure. Mitigation is the diff-reconciliation invariant + size preflight; without them the feature's flagship artifact can be a half-document that looks complete.
  2. The pointer inverts write-once and provenance direction for illusory benefit (Q2). Prefer file-only; reverse the pointer into changes.json.
  3. The title join key is not unique and the completeness check inherits that (Q1). Code-confirmed, not theoretical; amend already silently mis-joins on duplicates.
  4. caveats doesn't exist — the third container is dissent, which isn't "resolvable" (Q3). The resolves enum and completeness scope are specified against the wrong container.
  5. Endorsement can manufacture consensus on parse failure, and can't object to unresolved (Q6/Q7). ABSTAIN-default and objectable-unresolved close both.

3. Recommended execution sequence

Fold these into the roadmap's D9+ / Phase 2–4 gating:

  1. Pre-work (blocking): decide Q2 (recommend file-only) and reconcile the caveats→dissent naming; both change the changes@1 schema before a line is written.
  2. Phase 2a — join integrity: add title-uniqueness enforcement at synthesis + ambiguous-resolves refusal in the changes validator. Ship this even if it lands slightly ahead as a small amend-bug fix.
  3. Phase 2b — revision seat: generalize the synthesizer spawn path (seat-choice synthesizer.py:205-233, two-attempt retry, skeleton merge, DATA-fence) into a shared helper; add the source-size preflight and the --source-type resolution (Q4).
  4. Phase 2c — changes.json + reconciliation: author changes@1 + its validator; implement the diff-reconciliation gate (edits ↔ mechanical get_opcodes diff) — this is the integrity spine and also the truncation guard.
  5. Phase 3 — artifacts: revised-draft.*, .patch (prose and code), then the word-level HTML redline as an isolated, golden-file-tested, non-release-gating module.
  6. Phase 4 — endorsement: ON-by-default, ABSTAIN-on-parse-failure, objectable unresolved; then release.
  7. Adversarial pass before merge (per project convention): re-run this board + finders on the built diff, with attention to the reconciliation gate and truncation handling.

4. Invariants and guardrails

  • INV-1 (integrity spine): changes.json.edits[] must reconcile 1:1-or-better with the mechanical original→revised diff. Every diff hunk ↔ ≥1 edit; every locator ⊂ a hunk. Reconciliation failure ⇒ reject path (retry, then *-rejected + exit 0 / --strict-exit 4). This one invariant also makes truncation a loud failure.
  • INV-2: the revision step never opens verdict.json (Q2a). A revision failure is then structurally incapable of corrupting the judgment record — matching the cli.py:554-613 "don't destroy prior good state" discipline.
  • INV-3: finding titles are unique within a list before any resolves join exists.
  • INV-4: endorsement silence ⇒ ABSTAIN, rendered distinctly from ENDORSE. No model reply is ever interpreted as endorsement.
  • INV-5: model authorship stays confined to summary/resolves/note; conductor owns n, shas, statuses, locator validation, provenance (PROTECTED_SKELETON_KEYS + LIFECYCLE_KEYS, synthesizer.py:192-203).
  • INV-6 (D5): no-flag default run stays byte-identical; assert with the existing golden fixtures plus a new "revision path absent ⇒ zero new bytes" test.
  • INV-7 (D6): source is never written; the revision step has no write path to the source, only to the run dir.

5. Risks, stale assumptions, missing evidence

  • Stale assumption — "titles are stable ⇒ safe join": stable ≠ unique (INV-3). Code-confirmed.

Keep full-text + mechanical diff as the source of truth; a seat-emitted patch is at most an optimization to validate, never to trust.

  • Under-specified — dissent's role: if dissent stays out of resolves, does the revised draft acknowledge dissents at all? Decide explicitly rather than by omission.

A header means applying it prepends board metadata to the user's document — and for code, revised-draft.<ext> with a header is a syntax-corrupt or semantically-wrong file the moment it's saved. Verdict: CAUTION

Confidence: HIGH

Change Factor: This verdict would improve to SHIP if the latency-heavy sequential endorsement loop proposed in Q6 is optimized to be off-by-default or parallelized to avoid stalling the main user loop.

---

2. Strongest Objections & Operational Risks

  • The Sequential Endorsement Bottleneck (Q6): Spawning additional review runs sequentially across all non-revision seats adds immense execution latency (up to 40-60% longer runs in a standard 3-seat board). This is highly disruptive to active CLI developer workflows.
  • Post-Synthesis Write-Path Reopening (Q2): Modifying verdict.json after the synthesizer has finalized it violates transactional single-write guarantees. This exposes the file to dirty writes if concurrent actions (like human amend or CI validation gates) run simultaneously.
  • Line-Range Locator Drift (Q3): Relying solely on line-range numbers without original context blocks means patches will break silently if a human makes minor local file modifications before applying the board's proposed changes.

---

3. Rulings on the Eight Questions (Q1–Q8)

Q1 — Edit Identity: Finding ids or exact titles?

  • Ruling: (a) Key by exact finding title.
  • Rationale: Titles are fully functional in the codebase today, as seen in amend --on and --revise delta matching. A complete schema evolution to @3 just to support unique IDs is unnecessary overhead and causes backward compatibility churn. Finding titles are stable during the short minutes between synthesis and revision.

Q2 — Where does the mapping live: changes.json file vs. the reserved changes key?

  • Ruling: (a) Separate changes.json artifact only. (Dissenting from Strawman (b))
  • Rationale: Once verdict.json is written by the synthesizer and verified, the write path should be closed. We must avoid reopening verdict.json post-synthesis. A sovereign, self-contained changes.json maintains pipeline isolation and protects the integrity of the canonical verdict.

Q3 — changes.json Shape & Locator Mechanics

  • Ruling: Include line ranges + strict before/after context snippets. (Dissenting from line-only)
  • Rationale: Pure line-range locators are too fragile. Including short before/after blocks allows safety checks to confirm structural context matches before applying mutations. A complete validation check must be enforced: every blocker in the verdict must either map to an applied edit or be declared in the unresolved conflict array.

Q4 — Prose vs. Code: What decides redline vs. patch?

  • Ruling: (c) Heuristic default + flag override.
  • Rationale: Fully automatic detection fails on ambiguous extensions, and fully manual flags add friction. Heuristic detection for common extensions, combined with mandatory --source-type flags for stdin/URLs, recorded in the recipe, balances exact CLI ergonomics with reproducible replays.

Q5 — One spawn or two: How does the revision seat return text + mapping?

  • Ruling: (a) One spawn with two fenced sections.
  • Rationale: A single spawn guarantees transactional consistency. Splicing text and JSON metadata in a single response prevents text-mapping drift and avoids doubling input token volume, operational latency, and API costs.

Q6 — Endorsement pass: Shape and default

  • Ruling: (ii) OFF by default, opt-in via --endorse. (Dissenting from Strawman (i))
  • Rationale: Running an endorsement pass on every revision run introduces substantial latency overhead. Given that applying the revision is always a human decision under D6, the feature's primary integrity is preserved without forcing a mandatory multi-seat consensus loop. The endorsement pass must be an explicit, optional opt-in.

Q7 — Failure posture when findings conflict

  • Ruling: (c) Apply non-conflicting edits; surface conflicts in unresolved.
  • Rationale: Graceful degradation is a core operational standard. If the board produces conflicting recommendations, the revision seat should resolve what it can and log conflicts in unresolved rather than failing the run or choosing a side silently. It should not affect the final exit code.

Q8 — Redline artifacts: Exactly what does P3 render?

  • Ruling: Ship as specified by the strawman.
  • Rationale: Outputting clean draft files alongside unified .patch files (via standard difflib) and word-level HTML rendering is clean and idiomatic. Non-standard markdown formats (like CriticMarkup) add unnecessary rendering and parsing complexity.

---

4. Recommended Execution Sequence

  1. Phase 1 (CLI & Configuration): Add parsing and validation for --output revised-draft and --source-type.
  2. Phase 2 (Draft Generation): Build single-spawn prompts, fence parsers, and validation rules for changes.json.
  3. Phase 3 (Diff Engine): Wire difflib to generate unified patches for code and inline word-level HTML renders for prose.
  4. Phase 4 (Endorsement Pass): Add the optional, concurrent endorsement step, recording responses in changes.json.

---

5. Invariants and Guardrails

  • D5/D6 Compliance: Source files must never be modified by the CLI. The revision remains an opt-in artifact.
  • Fail-Safe Exit: Revision/endorsement failures must never discard successful round reviews or delete valid synthesized verdict.json files.
  • Egress Bounds: Operation must utilize existing egress consent hashes; no new external data categories may be introduced.

  • Target Workspace State: Developers may modify local files between running a review and applying a patch, causing standard patch tools to fail.

Consensus blockers — must fix before ship

  1. Title-only edit identity is ambiguous by construction (Q1)
    All three seats moved off pure title matching for the revision's `resolves` join key. Unlike the human-in-the-loop `amend`, `changes.json` is machine provenance with no human to disambiguate, and there is no duplicate-title guard anywhere: `_finding_titles` appends every finding title unconditionally and covers only blockers+concerns.   2. `caveats` is a phantom container the schema must be fixed against (Q3)
    The brief and Q3 treat `caveats[]` as structured, resolvable findings (blockers/concerns/caveats), but the implementation's evidence containers are blockers/dissent/concerns, and caveats are plain string arrays with no titles or evidence — un-resolvable by construction. Every `resolves.list` enum, the completeness rule, and the best-effort carve-out inherit this error.   3. Silent truncation ships a corrupted "board-endorsed" copy (Q5)
    Claude's highest-severity failure and a top objection for all three seats: a single-spawn full-file rewrite of a large source can truncate and produce a plausible-looking short reply that gets written as the "fixed copy." The retry set only catches timeout, not truncation. Defense must be layered: strict fence parsing (an opened DATA fence with no matching close ⇒ reject/retry), a mechanical original→revised diff reconciliation (every edit locator maps to a real diff hunk 1:1; status computed from the diff, never model-asserted), and a source-size preflight that refuses oversized `--output revised-draft` loudly rather than truncating. Gemini proposes emitting the JSON mapping first and the draft second so a missing closing fence is caught mechanically.
  4. Metadata header inside the clean revised draft corrupts the applied artifact (Q8)
    The strawman's `revised-draft.*` carries a header (run title, seat, unresolved count). But that artifact is the thing a human applies — a header prepends board metadata to the user's document, and for code produces a syntax-corrupt or semantically-wrong file the moment it is saved. All three seats agree the draft must be byte-clean; metadata belongs in `changes.json`, the HTML handoff, and the run card, never in the applied file. Its sha256 in `changes.json.revised` must match the file on disk exactly.
  5. Sequential endorsement makes the default path slow and self-defeating (Q6)
    The board converged on endorsement ON by default (otherwise "board-endorsed" becomes marketing and one seat's rewrite wears the whole board's name), but it MUST be dispatched concurrently. Gemini's round-1 "+40–60%" is a spawn/token-cost figure miscast as latency; the endorsement pass is a dependency-free per-seat fan-out — structurally a review round — and rounds already fan out concurrently, so reused its wall-clock cost is ~one extra round, not N sequential spawns. `--no-endorse` remains the opt-out for the genuine token-cost axis.
